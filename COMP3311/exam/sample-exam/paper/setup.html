These notes describe the environment available to you in this exam. Reading them carefully could potentially save you much time in completing the questions.

The following scripts are available for you (simply type their name):

Script What it does
sql Runs SQLite on the footy.db file with columns and headings.
Could be handy for interactive SQLite sessions.
check qX Checks whether your qX view produces the expected result.
Assumes that the view is already loaded into your database
submit qX.suffix Submits the file cointaining your answer to question qX
Examples of usage:   submit q2.sql,   submit q10.txt

Hints on how to use your environment with minimal typing 

# load a view into your database
$ sql < q2.sql
# any load errors will be displayed here

# test the view you just loaded
$ check q2
# an indication of whether your view works is shown here

# if it's correct, then submit it
$ submit q2.sql

The above scenario applies if you're an SQL master, but if you have errors then you'll need to run an interactive session with the database and see which parts of your SQL aren't working. And you'll probably run such a session while you're developing your view anyway.

q7.* No template is supplied for this question

q12.txt Template for relational algebra in Question 12 (written)

The qX.sql and qX.txt are provided to save you some typing. Files with a sql suffix are part of the practical component of this exam and you should testthat the code in them works before submitting. Files with a .txt suffix are part of the written component of the exam, and, even if they contain code, do not require testing before submission. Of course, you should read them carefully before submitting.

Also copied to your directory will be a collection of files for testing:

File Contents
tests The test queries used by the check script
q*.expected Expected outputs for each of the Prac questions
To facilitate testing, these files are in default SQLite format
and thus have no headers or column-isation

If you accidentally remove any of the above files, you can get a fresh copy via the command: 

reload fileName

This command will not overwrite any existing copy of a file called fileName.

Be careful not to remove other files. You cannot restore them. It is a useful strategy to submit your work every so often. You are allowed to make multiple submissions; only the latest one will be marked.

The check script will create other files in the directory when it runs its tests. You can examine these to help with testing.
